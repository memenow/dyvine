Control-flow graph

Decision points and terminals inside the livestream URL resolver, the path-jail enforcement, and the bulk-download pagination loop.

Updated 2026-05-10

Two flows dominate the control-flow story: the multi-stage livestream URL resolver (LivestreamService.download_stream) and the bounded pagination loop shared by user content and bulk-post downloads. Both have explicit terminals so an adversarial input cannot pin a worker indefinitely. 

flowchart TD
    Start[Request Entry] --> Auth{require_api_key}
    Auth -->|No| ErrAuth[401 Unauthorized]
    Auth -->|Yes| Validate[Validate Pydantic body]
    Validate -->|Invalid| Err422[422 Validation]
    Validate -->|Valid| Route{Route Type}

    Route -->|Livestream| ParseURL[_parse_url]
    Route -->|Posts bulk| StartBulk[start_bulk_download]
    Route -->|User download| StartUser[start_download]

    ParseURL --> ResolveID{Resolve webcast_id}
    ResolveID -->|Numeric ID| DirectID[Use as webcast_id]
    ResolveID -->|URL segment| ExtractID[Use last URL segment]
    ResolveID -->|Profile URL| ProfileLookup[_resolve_from_profile]
    ResolveID -->|Unresolvable| LiveErr[LivestreamError -> 404]

    ProfileLookup --> IsLive{room_id present?}
    IsLive -->|No| LiveErr
    IsLive -->|Yes| StreamMaps[_stream_map_from_room_data]

    DirectID --> LoadFilter[_load_live_filter]
    ExtractID --> LoadFilter
    StreamMaps --> LoadFilter

    LoadFilter --> FilterOK{Filter loaded?}
    FilterOK -->|No + no profile| LiveErr
    FilterOK -->|Yes| StatusCheck{status == 2?}

    StatusCheck -->|No| LiveErr
    StatusCheck -->|Yes| Resolve[_resolve_streams]
    Resolve --> Pick[_select_stream_url]
    Pick -->|No match| LiveErr
    Pick -->|Found| Jail[resolve_within_root + ensure_within_root]

    Jail -->|Escapes root| Err422
    Jail -->|Symlink segment| Err422
    Jail -->|Within jail| Dedupe{Active dedupe lock entry?}
    Dedupe -->|Already running| LiveErr
    Dedupe -->|None| CreateOp[OperationStore.create_operation]
    CreateOp --> Spawn[BackgroundTaskRegistry.spawn]
    Spawn --> Pending[Return 202 with operation_id]

    StartBulk --> ValidateUser{Profile exists?}
    ValidateUser -->|No| ErrUser[UserNotFoundError -> 404]
    ValidateUser -->|Yes| BulkOp[OperationStore.create_operation]
    BulkOp --> BulkSpawn[Spawn _run_bulk_download]
    BulkSpawn --> BulkLoop[for page in pages]
    BulkLoop --> CursorCheck{cursor == previous?}
    CursorCheck -->|Yes| EndPagination[End pagination
terminal partial/completed]
    CursorCheck -->|No| FetchBatch[_fetch_posts_batch]
    FetchBatch -->|Empty| EndPagination
    FetchBatch -->|Has aweme_list| Process[_process_posts_batch]
    Process --> R2Path{R2 enabled?}
    R2Path -->|Yes| R2Upload[Upload to R2]
    R2Path -->|No| LocalOnly[Keep on disk]
    R2Upload --> Update[update_operation progress]
    LocalOnly --> Update
    Update --> Bound{page_count > max_pages?}
    Bound -->|Yes| EndPagination
    Bound -->|No| BulkLoop
    EndPagination --> Final[Persist terminal status]

Notes

  • The path-safety jail (core.path_safety) walks unresolved segments before Path.resolve() follows symlinks, then re-checks after mkdir(parents=True) to close the residual TOCTOU window.
  • The bulk-download page_count > max_pages guard uses core.pagination constants (PAGE_MULTIPLIER, PAGE_SLACK, MAX_PAGES_FALLBACK) so a sticky upstream cursor cannot pin a worker even when each page is non-empty.
  • The dedupe lock window in LivestreamService.download_stream spans the entire check-then-register sequence so two concurrent callers for the same room cannot both schedule a download.
Implementations live in src/dyvine/services/livestreams.py and src/dyvine/services/posts.py. The pagination constants are documented in src/dyvine/core/pagination.py.
