End-to-end request flowchart

High-level view of how a request fans out across the three router domains, the operation store, and Cloudflare R2.

Updated 2026-05-10

The flowchart below traces a typical request from the FastAPI router through validation, domain routing, external API calls, optional R2 storage, and the operation store that backs every asynchronous workflow. 

flowchart TD
    A[Client Request] --> Auth{X-API-Key valid?}
    Auth -->|No| Reject[401 Unauthorized]
    Auth -->|Yes| Mid[Correlation middleware]
    Mid --> B[FastAPI Router]
    B --> C{Valid Route?}
    C -->|No| E[Error Handler]
    C -->|Yes| D{Which Domain?}

    D -->|Users| U[UserService]
    D -->|Posts| P[PostService]
    D -->|Livestreams| L[LivestreamService]

    U --> F[Douyin API via f2]
    P --> F
    L --> F

    F --> G{Upstream OK?}
    G -->|No| E
    G -->|Yes| H[Domain logic]

    H --> S{Storage Required?}
    S -->|Yes| R[Cloudflare R2]
    S -->|No| I[Response Formatter]
    R --> I

    L --> LD{Download Requested?}
    P --> PD{Bulk download?}
    U --> UD{Bulk download?}

    LD -->|Yes| OP[(OperationStore SQLite)]
    PD -->|Yes| OP
    UD -->|Yes| OP

    OP --> TR[BackgroundTaskRegistry]
    TR --> BG[Background download task]
    BG --> R
    BG --> OP
    OP --> I

    I --> J[Client Response]
    E --> J
    Reject --> J

Notes

  • The require_api_key dependency is mounted at every router so handlers cannot opt out by accident; bypass requires SECURITY_REQUIRE_API_KEY=false.
  • The correlation middleware in main.request_middleware assigns a fresh UUID per request, mirrors it back via X-Correlation-ID, and pins it into the contextvars-based logger.
  • Every "Bulk download" entry returns immediately with a persisted operation; the actual fetch + R2 upload runs on a tracked background task drained at lifespan shutdown.
Diagram source mirrors src/dyvine/main.py, the three router modules, and core/dependencies.py. View the underlying calls in the call graph.
