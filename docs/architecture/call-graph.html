Call graph

Primary call relationships among the routers, services, and infrastructure helpers.

Last reviewed: 2026-05-29

The call graph below summarises the principal calls that flow from the FastAPI entry point into the service layer and the persistence/storage helpers. It is intentionally lossy — see the source modules and control-flow graph for the exhaustive paths.

Notes

  • Every long-running fan-out is mediated by BackgroundTaskRegistry.spawn via the spawn_or_fallback helper, so the lifespan can drain in-flight work before tearing down the executor pools.
  • OperationStore public methods are coroutines that dispatch onto the dedicated sqlite_executor; readers use per-thread connections so progress polls run in parallel with writes.
  • The _list_objects_sync path uses the shared r2_head_executor when one is attached so several concurrent listings cannot each spawn their own pool.
